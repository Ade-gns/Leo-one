PHASE 1 — Architecture Globale, Protocoles Sécurisés & Cycle de Vie de l'Agent

Stack : Agent C | Backend Go | Frontend React/TS | PostgreSQL + TimescaleDB

Protocole Agent ↔ Backend : WebSocket Secure (WSS) + mTLS

Document généré le 24 juin 2026

1. Vue d'Ensemble du Système — Macro-Architecture

Le système Leo-One est composé de trois couches principales communiquant exclusivement via TLS 1.3 :

  • Agents C — démons légers déployés sur chaque machine cible (Windows, Linux, macOS)
  • Backend Go — serveur central traitant les données, exposant les API REST et le gateway agent
  • Frontend React/TS — interface web temps réel pour les techniciens MSP

2. Modèle Multi-Tenant (Mode MSP)

Le système est organisé en trois niveaux d'isolation. Chaque entité est systématiquement associée à un tenant_id cryptographique, validé en middleware à chaque requête.

Platform (Leo-One SaaS / On-Prem)
│
├── Tenant A  (ex: "Cabinet Dupont IT")
│   ├── Workspace "Paris"
│   │   ├── Agent-001  (Windows Server)
│   │   └── Agent-002  (Windows 10)
│   └── Workspace "Lyon"
│       └── Agent-003  (Ubuntu 22.04)
│
└── Tenant B  (ex: "Mairie de Bordeaux")
    └── Workspace "Hôtel de Ville"
        └── Agent-004  (Windows 11)
Règle d'isolation stricte : il n'existe aucun chemin de code cross-tenant. Toute requête sans tenant_id valide et cryptographiquement signé est rejetée par le middleware avant d'atteindre la logique métier.

3. Flux Réseau Sécurisés

3.1 — Modèle de Sécurité : Mutual TLS (mTLS)

Protocole retenu : WebSocket Secure (WSS) sur TLS 1.3 avec authentification mutuelle par certificat.

┌──────────────────────────────────────────────────────────────────┐
│  MODÈLE DE SÉCURITÉ TLS — Mutual TLS (mTLS)                     │
│                                                                  │
│  Autorité de Certification (CA) interne, gérée par le Backend   │
│                                                                  │
│  ┌────────────────┐                  ┌────────────────────────┐  │
│  │   Agent C      │                  │   Backend Go           │  │
│  │                │                  │                        │  │
│  │ cert_agent.pem │── présente ────> │ valide via CA interne  │  │
│  │ key_agent.pem  │                  │                        │  │
│  │                │ <── présente ─── │ cert_server.pem        │  │
│  │ valide via     │                  │                        │  │
│  │ CA pinné       │                  │                        │  │
│  │ (embarqué)     │                  │                        │  │
│  └────────────────┘                  └────────────────────────┘  │
│                                                                  │
│  • TLS 1.3 minimum                                               │
│  • Suite : TLS_AES_256_GCM_SHA384                               │
│  • Certificate pinning côté agent : hash SHA-256 du cert CA     │
│    embarqué au moment de l'enrollment                            │
└──────────────────────────────────────────────────────────────────┘

3.3 — Enveloppe de Message JSON (format canonique)

Tous les messages transitent dans des frames WebSocket de type TEXT avec cette enveloppe commune :

{
  "v":    1,
  "type": "METRICS",
  "id":   "01J2K...",
  "ts":   1719230400123,
  "body": { ... }
}
ChampTypeDescription
vintVersion du protocole (actuellement 1)
typestringType de message (voir catalogue ci-dessous)
idstringUUID v7 généré par l'émetteur (pour ACK / corrélation)
tsint64Timestamp Unix en millisecondes (émetteur)
bodyobjectPayload spécifique au type de message

3.4 — Catalogue des Types de Messages

Agent → Backend :

typeFréquenceDescription
HELLOUne fois à la connexionIdentification, version, OS
HEARTBEATToutes les 30sPreuve de vie, numéro de séquence
METRICSToutes les 60sCPU / RAM / Disk / Net
INVENTORYSur demande ou toutes les 24hInventaire HW/SW complet
CMD_RESULTAprès exécutionRésultat d'une commande (stdout/stderr/exit_code)
LOGSur événementErreur ou événement notable de l'agent

Backend → Agent :

typeDescription
HELLO_ACKConfirmation de session + paramètres serveur (fréquences, config)
EXEC_SCRIPTExécuter un script PS1 / Bash / CMD avec timeout
INSTALL_PKGInstaller un paquet (MSI / DEB / RPM) en mode silencieux
COLLECT_INVENTORYDéclencher un inventaire immédiat
REBOOTRedémarrage planifié ou immédiat
PINGTest de latence — l'agent répond PONG
CONFIG_UPDATEMise à jour des paramètres de l'agent (ex : fréquence métriques)

Sécurité de l'enrollment : le enrollment_token est un JWT signé, à usage unique (invalidé côté backend dès la première utilisation), avec TTL de 24h. Il ne peut pas servir à enregistrer deux agents différents.

4.4 — Stockage Local de l'Agent

Windows : C:\ProgramData\LeoOne\
├── agent.conf            ← config permanente (agent_id, tenant_id, endpoint)
├── certs\
│   ├── client_cert.pem   ← certificat signé par le CA backend
│   ├── client_key.pem    ← clé privée locale (permissions restreintes)
│   └── ca.fingerprint    ← hash SHA-256 du CA pour certificate pinning
└── logs\
    └── agent.log         ← rotation 7 jours, 10 MB max par fichier

Linux / macOS : /opt/leo-one/  (même structure)
Sécurité locale : client_key.pem est lisible uniquement par l'utilisateur système dédié (leo-agent sur Linux/macOS, NT AUTHORITY\SYSTEM sur Windows). Les permissions sont vérifiées au démarrage — si elles sont incorrectes, l'agent refuse de démarrer et écrit une erreur critique dans le log.
